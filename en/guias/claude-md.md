# How to create an effective CLAUDE.md

The `CLAUDE.md` file is the **constitution** of your collaboration with AI. It defines who the AI is, how it behaves, and what rules it follows. Claude Code reads this file automatically at the start of every conversation.

> **Note:** In this framework, `CLAUDE.md` is the identity file for **your** AI -- generated by `/comece-por-aqui`. The framework documentation lives separately in `CLAUDE-IC.md`.

## Why this matters

Without a CLAUDE.md, every conversation starts from scratch. The AI has no identity, doesn't remember your preferences, and doesn't understand your project's conventions. You'll spend the first 10 minutes of every session explaining yourself all over again.

With a well-crafted CLAUDE.md, the AI arrives **ready** -- with personality, rules, and context already loaded.

## Main sections

### 1. Identity

Give your AI a name and a role. This isn't cosmetic -- it creates a consistent interaction pattern.

```markdown
# Identity

I am **Leland Hawkins** -- a mentor, not an assistant.
```

**Key decisions:**
- **Name:** Makes the interaction feel intentional, not generic.
- **Role:** "Mentor", "collaborator", "architect" -- this shapes how the AI frames its responses. An "assistant" waits for orders. A "mentor" challenges you when you're wrong.

### 2. Personality

Define behavioral traits that activate in specific contexts. Don't describe a vague personality -- map traits to situations.

```markdown
## Personality

- **The Pragmatist** -- Activates during code review and architecture decisions.
- **The Provocateur** -- Activates during teaching moments and design discussions.
- **The Educator** -- Activates during explanations and breaking down complex concepts.
```

**Why context-specific traits work better:** A personality that's always "friendly and helpful" is noise. A personality that's "direct during code review, Socratic during design" is a tool.

### 3. Behavioral rules

Explicit rules that override default behavior. Be specific.

```markdown
## Behavioral Rules

- When coding: be efficient and precise. Personality lives in brief comments, not in slowing down work.
- When reviewing: be honest. If something is mediocre, say so.
- When the user is wrong: say it directly, then explain why.
- Never sacrifice productivity for personality.
```

**Common mistake:** Writing rules that are too vague ("be helpful"). Write rules you can verify ("never add docstrings to code you didn't modify").

### 4. Project conventions

Technical standards that apply to all work on the project.

```markdown
## Project Conventions

- All files, code, and comments: Portuguese (BR).
- File exchange: `troca/entrada` (user -> AI), `troca/saida` (AI -> user).
```

### 5. Current state

A brief summary of where the project stands. Update this at the end of each session.

```markdown
## Current State

- **Phase:** Setup complete. No code yet.
- **Next step:** Begin Layer 0 (Mathematics and Python fundamentals).
```

## Principles

1. **Be specific, not aspirational.** Don't write who you'd like the AI to be -- write rules it can actually follow.
2. **Less is more.** A 50-line CLAUDE.md that's precise beats a 500-line one that's vague.
3. **Update regularly.** A CLAUDE.md that doesn't reflect the current state of the project is worse than none at all.
4. **Test it.** Start a new conversation and see if the AI behaves as defined. If not, the CLAUDE.md needs adjustments.
5. **It's a living document.** Not a contract -- a constitution that evolves with the project.
